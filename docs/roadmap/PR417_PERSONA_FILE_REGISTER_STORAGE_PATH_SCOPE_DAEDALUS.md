# PR417 - Persona File Register Storage Path Scope

Owner: DAEDALUS
Opened by: MIMIR
Status: Open

## Why This Exists

PR416 fixed the leading PR415 signed-upload blocker by sanitizing the storage
object basename generated by:

```text
GET /persona-files/persona/:personaId/upload-url
```

ARGUS accepted that repair with one caveat: the historical register route still
trusts a caller-provided `storagePath`:

```text
POST /persona-files/persona/:personaId/register
```

The owner UI should register only the fresh storage path returned by the
matching upload-url request, but direct API callers should not be able to submit
arbitrary storage paths outside the authenticated owner/persona scope.

PR417 hardens that register boundary before MIMIR reopens hosted upload proof.

## Scope

Implement a narrow API repair in `apps/api/src/routes/persona-files.ts`.

The register route should reject any `storagePath` that is not a normal object
path under the authenticated owner and requested persona:

```text
<ownerUserId>/<personaId>/<object-basename>
```

Requirements:

- Accept valid paths returned by the upload-url route.
- Reject paths outside the current owner prefix.
- Reject paths outside the current persona prefix.
- Reject leading slash, backslash, URL-like, query/hash, empty basename, `..`,
  or traversal-shaped input.
- Do not echo raw rejected paths in owner-visible errors.
- Preserve original owner-visible filenames in file registration/readback.
- Preserve valid duplicate/idempotency behavior keyed to the returned
  `storagePath`.
- Keep quota accounting and best-effort cleanup behavior intact.

## Acceptance Criteria

- `POST /persona-files/persona/:personaId/register` rejects out-of-scope
  `storagePath` before file insert or import job creation.
- The rejection response is sanitized and does not include the submitted raw
  path.
- Valid upload-url-generated paths still register successfully.
- Existing same-path duplicate/idempotent behavior still works for valid paths.
- Same basename under another owner or persona is rejected.
- Traversal-like and URL-like input is rejected.
- Storage quota reserve/release behavior remains balanced on failure.

## Focused Tests

Add or update `test:storage` coverage for:

- valid sanitized upload-url path -> register succeeds;
- PR416-style sanitized proof filename -> register succeeds;
- `otherOwner/<personaId>/file.txt` -> register rejects;
- `<ownerUserId>/otherPersona/file.txt` -> register rejects;
- `<ownerUserId>/<personaId>/../other/file.txt` -> register rejects;
- leading slash, backslash, URL-like, query/hash, and empty basename shapes
  reject;
- rejected responses do not echo the raw path;
- duplicate/idempotency behavior for a valid returned path is unchanged.
