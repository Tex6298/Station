# PR136 2C Observed Runtime Dedicated-Key Staging Smoke

Status: Blocked by dedicated smoke Developer Space creation limit on
2026-06-21; MIMIR decision needed.

## Why This Lane

PR134 proved the Agents Observe live-send path is explicit, guarded, dry by
default, and testable with mocked transport. PR135 added named Developer Space
ingestion keys so smoke/operator keys can be created without rotating unrelated
active integration keys.

The next step is the real PR130-style staging smoke, but only through the new
named-key route and only with strict no-secret handling. This lane should prove
the external-runtime/operator path end to end against staging if auth/config is
available; otherwise it should produce a precise missing-config report without
mutating live keys or asking for broad app env.

## Scope

- Target the Railway web/API staging URL already used by Station:
  `https://stationweb-production.up.railway.app` unless local env explicitly
  overrides it.
- Use a dedicated smoke Developer Space if one already exists; otherwise create
  one only if the authenticated owner API path and test account are available.
- Create a named ingestion key with the new route:
  `POST /developer-spaces/:id/ingestion-keys`.
- Label the key clearly, for example `PR136 observed-runtime smoke`.
- Do not use legacy `POST /developer-spaces/:id/api-key`, because it revokes
  unrelated active keys.
- Use the one-time raw key only in-process as external sender/operator config.
  Do not write it to `.env`, docs, logs, commits, or Railway variables.
- Run the Agents Observe guarded live-send path with:
  - `STATION_API_URL`;
  - the one-time named key as `STATION_DEVELOPER_KEY`;
  - a unique `STATION_OBSERVED_RUNTIME_WEBHOOK_ID`;
  - `STATION_OBSERVED_RUNTIME_SIGNING_SECRET` only if the target Developer
    Space has an active dedicated observed-runtime signing secret that requires
    it.
- Verify the smoke result through safe API/readback evidence:
  - accepted send or bounded rejection;
  - no duplicate/replay/conflict surprise;
  - public-safe readback does not expose raw prompt, command body, paths,
    tokens, tool payloads, terminal output, key material, signing material, or
    webhook id;
  - owner readback, if used, is bounded and does not print secrets.
- Revoke the smoke named key at the end if it was created solely for this lane,
  unless MIMIR explicitly decides it should remain as a reusable staging smoke
  key. If revoked, prove the revoke was targeted and did not revoke unrelated
  active keys.
- Record exact config status without printing secret values.

## If Config/Auth Is Missing

Do not stall silently. Produce a precise missing-config report and wake MIMIR.

Classify missing inputs like this:

- `STATION_API_URL`: app/API target URL; local `.env` may already have it.
- Auth/session/test-account credentials: Station owner login needed to create or
  select the smoke Developer Space and named key.
- Named key raw value: generated by Station via
  `POST /developer-spaces/:id/ingestion-keys`; external sender/operator secret,
  not general Station backend env.
- `STATION_OBSERVED_RUNTIME_WEBHOOK_ID`: generated unique idempotency/delivery
  id for the smoke.
- `STATION_OBSERVED_RUNTIME_SIGNING_SECRET`: only needed when the target
  Developer Space has an active dedicated observed-runtime signing secret and
  the client-side send path must sign with it.

## Acceptance

- Smoke uses the named-key route and does not rotate unrelated active keys.
- No secret values are printed, committed, written to `.env`, or included in
  docs.
- The live send either succeeds with safe readback proof or fails with a bounded
  classified error that identifies the missing/invalid non-secret boundary.
- Replay/idempotency behavior is documented for the smoke webhook id.
- If a temporary smoke key is created and then revoked, targeted revoke is
  proven.
- If smoke cannot run, the handoff says exactly what is missing and why, without
  reclassifying `STATION_DEVELOPER_KEY` as general backend config.
